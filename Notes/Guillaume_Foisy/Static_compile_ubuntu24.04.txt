n’est pas une erreur de GTK, ni de PLplot, ni de gtk‑fortran.
C’est le linker qui te dit :

« Ton binaire PIE contient du code qui doit être modifié en mémoire → donc il n’est plus pure position‑independent code. »

Autrement dit :
une des bibliothèques Fortran que tu linkes n’est pas compilée en position‑independent code (-fPIC).

🎯 Pourquoi ça arrive avec gtk‑fortran + PLplot
GTK4 et gtk‑fortran sont normalement compilés avec -fPIC.
Mais PLplot Fortran (ou ton build local) est souvent compilé sans -fPIC.

Donc quand tu fais :

Code
gfortran gui.f90 -o gui `pkg-config --cflags --libs gtk-4-fortran plplot plplot-fortran`
tu obtiens un binaire PIE (par défaut sur Mageia), mais une lib statique ou partagée non‑PIC force le linker à créer des relocations dans .text.

✅ Solution 1 — Compiler PLplot Fortran avec -fPIC
Si tu as compilé PLplot toi‑même :

Code
cmake -DCMAKE_POSITION_INDEPENDENT_CODE=ON ..
make
sudo make install
ou ajouter dans l’environnement :

Code
export CFLAGS="-fPIC"
export CXXFLAGS="-fPIC"
export FFLAGS="-fPIC"
Puis recompiler PLplot.

✅ Solution 2 — Désactiver PIE pour ton binaire (simple, rapide)
Tu peux dire à gfortran : « ne fais pas un binaire PIE ».

Code
gfortran gui.f90 -o gui -no-pie \
    `pkg-config --cflags --libs gtk-4-fortran plplot plplot-fortran`
→ Plus de warning, ton programme fonctionne pareil.

✅ Solution 3 — Forcer le linker à accepter les relocations
Pas recommandé, mais possible :

Code
gfortran gui.f90 -o gui -Wl,-z,notext \
    `pkg-config --cflags --libs gtk-4-fortran plplot plplot-fortran`

#!/usr/bin/env tclsh

package require websocket   ;# tcllib
package require http
package require sha1
package require base64

# Callback quand un message arrive
proc onMessage {sock type msg} {
    puts "SERVER: received: $msg"
    if {$msg eq "ping"} {
        ::websocket::send $sock text "pong"
    } else {
        ::websocket::send $sock text "echo: $msg"
    }
}

# Callback connexion / fermeture (optionnel)
proc onConnect {sock} {
    puts "SERVER: client connected: $sock"
}
proc onClose {sock} {
    puts "SERVER: client closed: $sock"
}

# Démarre un serveur WebSocket sur port 8080
::websocket::server 8080 ::wsHandler

proc ::wsHandler {sock} {
    ::websocket::takeover $sock ::onMessage
    onConnect $sock
    ::websocket::configure $sock -closecb ::onClose
}

puts "SERVER: WebSocket listening on ws://localhost:8080/"

1. Choisir le toolchain et le modèle mémoire
Toolchain:  
DJGPP (GCC 32‑bit DPMI pour DOS) — le plus logique pour du C moderne + flat memory.

Modèle mémoire:  
flat 32‑bit protégé (DPMI), pas de segments 16‑bit tordus.

Objectif:  
Produire une lib statique libbearssl_dos.a linkable dans ton EXE DOS (Fortran ou C).

2. Adapter la configuration BearSSL au contexte DOS
Pas d’OS, pas de threads, pas de syscalls avancés.

Désactiver tout ce qui est inutile :

pas de file I/O interne

pas de sockets internes

pas de dépendance à select/poll

Garder uniquement :

TLS 1.2 client (au début)

ciphersuites modernes minimales (AES‑GCM + ECDHE)

X.509 basique (ou même PSK au début pour simplifier)

BearSSL est déjà pensé pour être sans malloc, sans OS, avec contextes fournis par l’appli — ça joue en ta faveur.

3. Abstraire les I/O réseau (intégration GF‑NetStack DOS)
BearSSL ne gère pas les sockets lui‑même — c’est toi qui fournis les I/O.

Créer une petite couche “transport” côté GF‑NetStack DOS :

net_read(void *ctx, unsigned char *buf, size_t len)

net_write(void *ctx, const unsigned char *buf, size_t len)

Cette couche utilise :

soit mTCP / wattcp

soit ton propre driver TCP GF‑NetStack DOS

BearSSL voit juste un “flux TCP fiable” — il s’en fiche que ce soit DOS.

4. Gérer l’entropie (RNG) sous DOS
Gros point sensible.

BearSSL a besoin d’un RNG cryptographique pour :

clés éphémères

nonces

handshakes TLS

Sous DOS, tu n’as pas /dev/urandom.

Options réalistes :

mixer :

timestamp haute résolution (PIT)

compteur d’instructions / TSC (si dispo)

timings clavier / disque / réseau

éventuellement stocker un seed sur disque et le ré‑mixer à chaque boot.

Tu implémentes un callback RNG que BearSSL utilise.

5. Gérer l’horloge et les certificats
TLS aime bien avoir une heure approximative pour valider les certificats.

Sous DOS :

utiliser l’horloge BIOS / DOS

ou accepter des certs “sans vérifier la date” (mode dev / rétro).

Tu peux commencer avec :

validation minimale (chaîne + CA)

ignorer la date dans un premier temps.

6. Compiler BearSSL en statique pour DOS
Récupérer les sources BearSSL.

Adapter le conf/Unix.mk ou faire un Makefile.dos :

CC = gcc (DJGPP)

CFLAGS = -O2 -march=i386 -fno-asynchronous-unwind-tables -fno-exceptions

-ffreestanding si tu veux être strict

Construire :

libbearssl_dos.a

7. Intégrer dans GF‑NetStack DOS
Côté GF‑NetStack DOS :

ajouter un module tls_bearssl_dos.c

exposer une API simple :

gf_tls_connect(host, port, params)

gf_tls_read(...)

gf_tls_write(...)

gf_tls_close(...)

Côté Fortran (ou autre) :

tu appelles GF‑NetStack comme d’habitude, TLS devient juste une option.

🎯 Takeaway
Oui : tu peux compiler GF‑NetStack en .a statique DOS depuis Ubuntu 24.04.
Mais tu dois utiliser un cross‑compiler DJGPP (i586‑pc‑msdosdjgpp‑gcc).

🧱 1. Pourquoi tu ne peux pas compiler directement avec GCC Ubuntu
Ubuntu 24.04 utilise :

ELF

glibc

Linux syscalls

modèle mémoire 64‑bit

FreeDOS utilise :

MZ EXE

DPMI 32‑bit (DJGPP)

aucun syscall Linux

aucun ELF

Donc tu dois compiler avec un toolchain DOS, pas avec GCC natif.

🟦 2. Le toolchain qu’il te faut : DJGPP cross‑compiler
Tu dois installer un cross‑compiler DJGPP pour Ubuntu :

Code
i586-pc-msdosdjgpp-gcc
i586-pc-msdosdjgpp-ar
i586-pc-msdosdjgpp-ranlib

--> sudo apt install build-essential unzip wget

--> sudo apt install binutils-djgpp

Confirme que les outils DJGPP sont bien reconnus.

--> i586-pc-msdosdjgpp-ar --version

--> i586-pc-msdosdjgpp-ranlib --version

--> i586-pc-msdosdjgpp-ld --version


Ce toolchain génère :

.o compatibles DOS

.a statiques DOS

.exe DOS 32‑bit DPMI

C’est EXACTEMENT ce qu’il te faut pour GF‑NetStack DOS.

🟩 3. Compilation de GF‑NetStack en lib statique DOS
Une fois le cross‑compiler installé, tu peux faire :

Code
i586-pc-msdosdjgpp-gcc -O2 -c gfnet_http.c -o gfnet_http.o
i586-pc-msdosdjgpp-gcc -O2 -c gfnet_ws.c -o gfnet_ws.o
i586-pc-msdosdjgpp-gcc -O2 -c gfnet_tls_stub.c -o gfnet_tls_stub.o
Puis créer la lib statique :

Code
i586-pc-msdosdjgpp-ar rcs libgfnetstack.a \
    gfnet_http.o gfnet_ws.o gfnet_tls_stub.o
Et voilà :
libgfnetstack.a → compatible FreeDOS 1.4

🟧 4. Linker ton programme Fortran DOS avec GF‑NetStack
Si tu utilises gfortran DJGPP (cross‑compiler) :

Code
i586-pc-msdosdjgpp-gfortran -O2 -o app.exe app.f90 libgfnetstack.a
Résultat :

un EXE DOS 32‑bit DPMI

statique

autonome

portable

compatible FreeDOS 1.4

utilisant GF‑NetStack DOS

🟥 5. Et BearSSL DOS dans tout ça?

Code
